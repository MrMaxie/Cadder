title: How to use
description: Use Cadder releases, the caddy shim, and the TUI without changing Cadder itself.
---

The release contains four files you normally care about:

- `caddy`
- `cadderd`
- `cadder-tui`
- `cadder.toml`

The `caddy` command is Cadder's shim. Use it for project `caddy run` workflows that should be coordinated by Cadder. The real Caddy binary stays separate and is configured through [cadder.toml](../cadder-toml/).

## Basic flow

1. Get the release for your operating system from [GitHub Releases](https://github.com/MrMaxie/Cadder/releases/latest).
2. Point Cadder at the real Caddy binary with [cadder.toml](../cadder-toml/).
3. Run your project through Cadder:

```sh
caddy run
```

4. Inspect the daemon when needed:

```sh
cadder-tui
```

## Direct or PATH-based use

Both styles are valid.

Use a direct path when you want to be explicit about which binary runs. Put the Cadder shim on `PATH` when you want `caddy run` to mean "run this project through Cadder" from any project directory.

The practical setup is:

- keep the real Caddy binary away from the `PATH` entry used for Cadder;
- configure its path in [cadder.toml](../cadder-toml/);
- expose Cadder's `caddy` shim globally only if that matches your workflow.

## Check the shim

This command confirms that `caddy` is Cadder's shim:

```sh
caddy --cadder-shim-info
```

If it prints real Caddy output instead, your shell is finding real Caddy before Cadder's shim.
