## Direct use

From the directory that contains Cadder:

```powershell
.\caddy run
.\cadder-tui
```

From any project directory where Cadder is on `PATH`:

```powershell
caddy run
cadder-tui
```

## cadder.toml

Point Cadder at the real Caddy executable in [cadder.toml](../../../user-guide/cadder-toml/). On Windows, literal TOML strings let you write normal backslashes.

## Optional user PATH setup

If you want `caddy run` to go through Cadder everywhere, put Cadder's shim before real Caddy on your user `PATH`. Keep the real Caddy binary somewhere else and reference it from `cadder.toml`.

## TUI diagnostics

```powershell
cadder-tui
```

Use the dashboard for daemon state, domains, logs, diagnostics, and activation toggles.

## IIS handoff

Use [Windows IIS handoff](../iis/) when a local IIS site needs to sit behind Cadder's Caddy front door.

Cadder does not require the whole daemon or TUI session to run as Administrator for this workflow. Discovery, planning, restore metadata, Caddy config updates, and normal TUI operations stay in the user context. Windows asks for administrator approval only when Cadder must mutate IIS bindings.

## Recover from common failures

- If Cadder cannot find real Caddy, check [cadder.toml](../../../user-guide/cadder-toml/) first.
- If `caddy run` starts real Caddy instead of Cadder, your shell is finding real Caddy before the Cadder shim.
- If one project needs a different real Caddy binary, put a project-local `cadder.toml` in that project directory.
- If an IIS handoff prompt is denied, retry the same row when ready. Cadder leaves non-IIS daemon and TUI operations usable.
