## Process roles

- `cadderd` is the per-user daemon. It owns registrations, local IPC, Caddyfile adaptation, effective config composition, the Cadder-owned real Caddy process, runtime diagnostics, and bounded log storage.
- `caddy` is the PATH-facing shim. It intentionally shadows Caddy for managed `caddy run` commands, starts or connects to `cadderd`, registers the caller's config, heartbeats while alive, and unregisters on exit.
- `cadder-tui` is the Ratatui/Crossterm UI. It connects to the daemon and shows overview state, entrypoints, grouped domains, per-domain logs, diagnostics, search, filtering, and activation toggles.
- The real Caddy runtime is an external binary. Cadder never embeds Caddy and must not recursively execute its own shim.

The release-facing command names are `cadderd`, `caddy`, and `cadder-tui`.

## Runtime model

Cadder uses per-user runtime paths from `directories::ProjectDirs`. `CADDER_RUNTIME_DIR` can override that path for tests and custom deployments.

The daemon owns:

- a lockfile guarded by `fs4`, which prevents multiple daemons for the same runtime directory;
- a local IPC socket name derived from the runtime directory;
- a generated effective Caddy JSON config file;
- daemon metadata and bounded in-memory state.

The daemon starts on demand from `caddy` and `cadder-tui` unless callers opt out. OS service installation is intentionally outside the current model.

## IPC boundary

IPC is versioned newline-delimited JSON over a per-user local socket through `interprocess`. Each message has an envelope:

```json
{ "protocolVersion": 1, "type": "query-state-request", "payload": { "requestId": "..." } }
```

Supported public messages include registration, unregister, heartbeat, state query, state subscription, entrypoint/domain activation changes, log query, and daemon shutdown.

Shim registrations are tied to the IPC session that created them. If the shim exits without sending unregister, pipe disconnect cleanup removes only registrations owned by that session.

## Caddy integration

Cadder resolves real Caddy through the layered model documented in [Real Caddy resolution](/reference/real-caddy-resolution/). The path fallback excludes the current executable and the known shim path from `CADDER_CADDY_SHIM_PATH`.

For each registration, Cadder runs:

```sh
caddy adapt --config <Caddyfile> --adapter <adapter>
```

The adapted JSON is inspected for HTTP host matchers. The adapter resolves real Caddy against the registration source working directory, so project-local `cadder.toml` files are honored for shim-driven `caddy run` from arbitrary directories.

Active domains are composed into a generated effective Caddy JSON document. Domain conflicts are reported before runtime reload. When no active domains remain, the daemon enters idle config/runtime state instead of reloading an empty active config.

Runtime operations start the owned real Caddy process with the generated config and reload it on later config changes. Captured stdout, stderr, and control events are stored in a bounded log store with redaction for token-like values.

## Workspace layout

- `crates/cadder-protocol`: shared DTOs, activation/runtime/log states, IPC envelopes, and request/response contracts.
- `crates/cadder-daemon`: runtime paths, daemon lock, local IPC server/client, registration state, Caddy integration, process runtime, and log store.
- `crates/cadderd`: daemon binary.
- `crates/cadder-shim`: package containing the PATH-facing `caddy` binary.
- `crates/cadder-tui`: terminal UI.
- `xtask`: repository validation and portable layout task runner.
- `docs/site`: this Astro Starlight documentation workspace.
