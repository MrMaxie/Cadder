title: Release process
description: Portable layout, archive naming, and GitHub Actions publishing.

Cadder releases publish portable binaries and documentation generated from source. They do not publish OS installers or mutate machine-global state.

## Rust CI

`.github/workflows/ci.yml` runs on pushes to `master`, pull requests targeting `master`, and manual dispatch. It validates the Rust workspace on Windows, Linux, and macOS with:

```sh
cargo fmt --check
cargo clippy --workspace --all-targets -- -D warnings
cargo test --workspace
```

The automated tests use repository fake Caddy fixtures. CI does not install or require a machine-global real Caddy binary.

The workflow also includes a Docker-enabled Ubuntu end-to-end job:

```sh
cargo build -p cadderd -p cadder-shim
cargo test -p cadder-daemon --features docker-e2e --test testcontainers_e2e -- --ignored --test-threads=1
```

That job runs compiled host Cadder binaries against real Caddy inside an official Testcontainers-managed Caddy container. It gates release readiness separately from the cross-platform fake-Caddy matrix so local `cargo test --workspace` remains Docker-free.

The same workflow also runs a dedicated coverage job on `windows-latest` with `stable-x86_64-pc-windows-msvc`. That job installs `cargo-llvm-cov 0.8.7` and runs the project-owned gate:

```sh
cargo run -p xtask -- coverage
```

The coverage gate writes `target/llvm-cov/coverage-summary.json` and fails if total line coverage is below 85% or coverage cannot be measured.

CI also runs the project-owned validation command on Ubuntu:

```sh
cargo run -p xtask -- check
```

## Portable artifact

Build the current platform layout with:

```sh
cargo run -p xtask -- dist --out <output-dir>
```

The layout must include:

- `cadderd`
- `cadder-tui`
- `caddy`
- `cadder.toml`

On Windows the executables include `.exe`.

Verify the layout with:

```sh
cargo run -p xtask -- verify-dist --dir <output-dir>
```

Build a versioned release archive with:

```sh
cargo run -p xtask -- package \
  --out artifacts \
  --version <version> \
  --platform <platform> \
  --target <rust-target>
```

Release archives are named:

- `cadder-<version>-windows-x64.zip`
- `cadder-<version>-linux-x64.tar.gz`
- `cadder-<version>-macos-x64.tar.gz`
- `cadder-<version>-macos-arm64.tar.gz`

Each archive has a neighboring `.sha256` file.

The release process does not edit `PATH`, install services, write shell profiles, replace package-manager shims, use WinUI, require Windows App SDK, create MSIX packages, use NuGet, or depend on the .NET SDK.

## Release publishing

`.github/workflows/release.yml` runs for tags that match `v*`. It rejects `v0.*` tags before publishing because Cadder does not publish GitHub Releases before 1.0.0. Manual dispatch runs package-only dry runs and does not create a GitHub Release.

For accepted release tags, it builds native portable archives on these GitHub-hosted runners:

- `windows-latest` for `x86_64-pc-windows-msvc`
- `ubuntu-latest` for `x86_64-unknown-linux-gnu`
- `macos-15-intel` for `x86_64-apple-darwin`
- `macos-15` for `aarch64-apple-darwin`

Before packaging starts, the release workflow reruns the Docker/Testcontainers end-to-end preflight on Ubuntu. The `package` job depends on that preflight, so tag publishing cannot create portable artifacts unless the real-Caddy container-backed suite passes.

The workflow uploads the archives as workflow artifacts, then publishes the same archive and checksum files to the matching GitHub Release with `gh release`.

## Documentation publishing

`.github/workflows/docs.yml` runs on pushes to `master` that affect documentation, pull requests targeting `master` that affect documentation, and manual dispatch. It:

1. Installs Bun.
2. Runs `bun install --frozen-lockfile` in `docs/site`.
3. Runs `bun run check`.
4. Runs `bun run build`.
5. Uploads `docs/site/dist` as a Pages artifact.
6. Deploys the artifact to GitHub Pages.

Repository Pages settings must use GitHub Actions as the publishing source. Generated documentation output remains outside git.

Production documentation is configured for `https://maxie.dev/Cadder/`.
