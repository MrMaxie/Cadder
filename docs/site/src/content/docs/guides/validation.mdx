Use Cargo from the repository root for Rust validation:

```sh
cargo fmt --check
cargo clippy --workspace --all-targets -- -D warnings
cargo test --workspace
cargo run -p xtask -- check
```

`cargo run -p xtask -- check` is the single repository validation command. Focused commands are useful while iterating, but full validation should pass before closeout.

## Docker end-to-end checks

The Testcontainers suite is separate from the default Rust validation path. It requires a running Docker daemon, the Docker CLI on `PATH`, and permission to pull the official `caddy:2.10.0-alpine` image. It does not require a host-global Caddy installation.

Build the host Cadder binaries first, then run the ignored Docker-required test target:

```sh
cargo build -p cadderd -p cadder-shim
cargo test -p cadder-daemon --features docker-e2e --test testcontainers_e2e -- --ignored --test-threads=1
```

The suite runs host `cadderd` and the PATH-facing `caddy` shim against real Caddy inside a disposable Testcontainers-managed container. It uses a unique `CADDER_RUNTIME_DIR`, bind-mounted temporary files, dynamic mapped ports, and automatic container cleanup. The existing `ipc_lifecycle` tests remain the fast fake-Caddy lifecycle coverage and continue to run under `cargo test --workspace`.

When Docker is unavailable, the suite stays skipped by default because the test target requires the `docker-e2e` feature and the test itself is ignored. The explicit command above should fail loudly if Docker or the Docker CLI cannot be reached.

## Documentation checks

Run documentation checks from `docs/site`:

```sh
bun install --frozen-lockfile
bun run check
bun run build
```

`bun run build` writes `docs/site/dist`. The generated directory is ignored and should not be committed.

## Manual TUI smoke check

Use the repository file `docs/verification/tui-smoke.md` after lifecycle or TUI-facing changes. Record the platform, terminal backend, runtime directory, Caddy command, and any rendering or daemon issue.

## Coverage

Cadder uses `cargo-llvm-cov 0.8.7` for Rust workspace coverage. On Windows, install the MSVC toolchain because GNU coverage can fail without the profiler runtime:

```sh
rustup toolchain install stable-x86_64-pc-windows-msvc --profile minimal --component llvm-tools-preview
cargo install cargo-llvm-cov --version 0.8.7 --locked
```

On Linux and macOS, install the active toolchain component and the same cargo subcommand:

```sh
rustup component add llvm-tools-preview
cargo install cargo-llvm-cov --version 0.8.7 --locked
```

Run the project coverage gate from the repository root:

```sh
cargo run -p xtask -- coverage
```

On Windows, the command runs `cargo +stable-x86_64-pc-windows-msvc llvm-cov --workspace --json --summary-only --fail-under-lines 85 --output-path target/llvm-cov/coverage-summary.json`. On other platforms, it runs `cargo llvm-cov --workspace --json --summary-only --fail-under-lines 85 --output-path target/llvm-cov/coverage-summary.json`. It fails when total line coverage is below 85% or when coverage cannot be measured, including when `cargo-llvm-cov` or the Windows MSVC toolchain is missing.

Use `cargo run -p xtask -- coverage --output <path>` to write the JSON summary to another path.

No project-specific coverage exclusions are configured. Future exclusions must be limited to generated, platform-gated, or intentionally untestable code and documented beside the `xtask coverage` command definition.
